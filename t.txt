# BuildX Implementation Plan

This document outlines the architecture and tasks for building **BuildX**, an outcome-based freelancing platform using a split frontend/backend architecture with Next.js and Express.

## Architecture Overview
- **Frontend (`/frontend`)**: Next.js App Router, styled with Tailwind CSS, deployed to Vercel.
- **Backend (`/backend`)**: Node.js & Express API, deployed to Render.
- **Database**: Neon PostgreSQL.
- **ORM**: Prisma.
- **Validation**: Zod.
- **Authentication**: JWT based (bcrypt for hashing).

## 1. Backend Setup (`/backend`)

### 1.1 Project Initialization
- Initialize a Node.js project.
- Install dependencies: `express`, `cors`, `dotenv`, `jsonwebtoken`, `bcrypt`, `zod`, `cookie-parser`.
- Install dev dependencies: `nodemon`, `typescript`, `@types/express`, etc. (We will use TypeScript for reliability).
- Initialize Prisma (`npx prisma init`) and configure the schema.

### 1.2 Prisma Schema

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

enum Role {
  FREELANCER
  CLIENT
  ADMIN
}

enum ProjectStatus {
  PENDING_APPROVAL
  OPEN
  STAGE_1_EVALUATION
  STAGE_2_OPEN
  STAGE_2_EVALUATION
  COMPLETED
}

model User {
  id                     String           @id @default(uuid())
  name                   String
  email                  String           @unique
  password               String
  role                   Role
  
  // Blostem Simulation (Synthetic Data)
  kyc_verified           Boolean          @default(false)
  account_balance        Float            @default(0)
  payment_capacity_score Int              @default(50)

  projects               Project[]        @relation("ClientProjects")
  submissions            Submission[]
  profileHistories       ProfileHistory[]
}

model Project {
  id              String         @id @default(uuid())
  title           String
  description     String
  budget          Float
  status          ProjectStatus  @default(PENDING_APPROVAL)
  client          User           @relation("ClientProjects", fields: [clientId], references: [id])
  clientId        String
  submissions     Submission[]
  profileHistory  ProfileHistory[]
}

model Submission {
  id           String    @id @default(uuid())
  project      Project   @relation(fields: [projectId], references: [id])
  projectId    String
  user         User      @relation(fields: [userId], references: [id])
  userId       String
  stage        Int       // 1 or 2
  mvpLink      String
  architecture String
  plan         String
  score        Int?      // 0-100
  status       String    @default("PENDING") // PENDING, SHORTLISTED, REJECTED, WINNER
  feedback     Feedback?
}

model Feedback {
  id           String     @id @default(uuid())
  submission   Submission @relation(fields: [submissionId], references: [id])
  submissionId String     @unique
  text         String
}

model ProfileHistory {
  id           String    @id @default(uuid())
  user         User      @relation(fields: [userId], references: [id])
  userId       String
  project      Project   @relation(fields: [projectId], references: [id])
  projectId    String
  stageReached Int
}
```

### 1.3 Core Backend APIs
1. **Auth (`/auth`)**:
   - `POST /auth/register`: Create user (Client or Freelancer)
   - `POST /auth/login`: Issue JWT token.
2. **Projects (`/projects`)**:
   - `GET /projects`: List approved projects.
   - `POST /projects`: Create project (Client lock escrow sim).
   - `POST /projects/approve`: Approve project (Admin).
3. **Submissions (`/submissions`)**:
   - `POST /submissions`: Submit Stage 1 / Stage 2 work. Runs rule-based scoring.
   - `GET /submissions?projectId=&stage=`: View submissions (Client).
4. **Evaluation**:
   - `POST /shortlist`: Client picks top users, generates generic feedback for rejected users. Updates Profiles.
   - `POST /select-winner`: Client picks winner, releases fake escrow payload.
5. **Profiles (`/profile`)**:
   - `GET /profile`: User history and tracking.

### 1.4 Middleware
- **Auth Middleware**: Parses JWT and sets `req.user`.
- **Role Middleware**: Enforces `Client`, `Freelancer`, or `Admin` access.
- **Zod Middleware**: Basic request body schemas.
